# Auto Merge PR

Run a comprehensive review of a GitHub pull request, post the result back to GitHub as a review, and either auto-merge (when no critical/important issues are found) or request changes.

---

## Step 0: Parse Arguments

`$ARGUMENTS` is the PR identifier â€” either:

- A PR number (e.g., `1234`)
- A full GitHub PR URL (e.g., `https://github.com/org/repo/pull/1234`)

If `$ARGUMENTS` is empty, abort with: `Usage: /auto-merge-pr <pr-number-or-url>`.

Use the value verbatim as `<pr>` in the `gh` calls below â€” `gh` accepts both forms.

---

## Step 1: Pre-flight Check

Verify the PR is reachable from the current working directory:

```bash
gh pr view <pr> --json number,title,isDraft,mergeable,baseRefName,headRefName,author,url,state
```

- If the call fails, abort and tell the user â€” they're likely in the wrong repo or not authenticated.
- Parse the JSON with `jq` and store the fields. Stop if `state != "OPEN"`.

Then determine whether the viewer authored the PR (matters in Step 6):

```bash
gh api user --jq .login
```

Set `is_self = (author.login == viewer_login)`. GitHub forbids `--approve` and `--request-changes` on your own PR, so when `is_self == true` both review paths in Step 6 must use `--comment`. (`gh pr merge --auto` works fine on self-authored PRs â€” only the review verbs are blocked.)

Save the current branch so we can restore it at the end:

```bash
git rev-parse --abbrev-ref HEAD
```

---

## Step 2: Check Out the PR

```bash
gh pr checkout <pr>
```

If checkout fails (dirty working tree, etc.), abort and report the error verbatim. Do NOT try to stash or force the checkout â€” let the user resolve it. A common cause is a stale local branch with the same name as the PR's head ref that has diverged from the remote; the user should delete the stale branch (`git branch -D <name>`) or rerun `gh pr checkout <pr> -f` manually.

---

## Step 3: Run the Review

Invoke the `pr-review-toolkit:review-pr` skill via the Skill tool with no extra arguments. This runs all applicable agents.

Capture the full markdown summary it produces â€” you'll need its findings in Step 4.

If the sub-skill errors, hangs, or returns no findings at all, do NOT proceed to Step 4 with an empty result â€” that would silently classify the PR as "clean" and queue a merge. Instead, abort with an error message naming the failure and skip to Step 7 for cleanup.

---

## Step 4: Classify Findings

Walk through every finding in the review output. Bucket each into one of three severities:

| Source label | Normalized severity |
|---|---|
| `critical`, `CRITICAL`, `must fix` | `critical` |
| `important`, `HIGH`, `should fix` | `important` |
| `suggestion`, `MEDIUM`, `LOW`, `nice to have`, `nit` | `suggestion` |

Compute counts: `critical_count`, `important_count`, `suggestion_count`.

Set the decision flag:

```
clean = (critical_count == 0 AND important_count == 0)
```

---

## Step 5: Build the GitHub Review Body

Assemble a single markdown body. Omit any section whose count is zero. Always wrap the suggestions list in a collapsed `<details>` block.

```markdown
## Automated PR Review

**Verdict:** <one of the canonical strings â€” see Step 6 for the full set, including self-PR `(posted as comment â€” ...)` suffixes>

### Summary
- Critical: <critical_count>
- Important: <important_count>
- Suggestions: <suggestion_count>

### Critical Issues
- [<agent>] <file>:<line> â€” <short description>

### Important Issues
- [<agent>] <file>:<line> â€” <short description>

<details>
<summary><suggestion_count> suggestions</summary>

- [<agent>] <file>:<line> â€” <short description>

</details>

_Generated by the `auto-merge-pr` skill._
```

Keep each finding line to one sentence. Don't paste full agent reports â€” link readers back to the local review by including `file:line` references.

---

## Step 6: Post to GitHub

Pass the body via a HEREDOC to preserve markdown formatting. Pick the path based on `clean`.

### 6a. Clean path (no critical/important findings)

First, decide whether the merge should be skipped. Check the JSON from Step 1:

- If `isDraft == true` â†’ set `merge_skip_reason = "PR is draft"` and `should_merge = false`.
- If `mergeable == "CONFLICTING"` â†’ set `merge_skip_reason = "PR has merge conflicts"` and `should_merge = false`.
- If `mergeable == "UNKNOWN"` â†’ fine to proceed; GitHub is still computing mergeability and `gh pr merge --auto` queues the merge until it's ready.
- Otherwise â†’ `should_merge = true`.

Pick the review action and verdict line:

- `is_self == false` and `should_merge == true` â†’ `--approve`, verdict "Approved & queued for merge".
- `is_self == false` and `should_merge == false` â†’ `--comment` (don't approve a draft/conflicting PR), verdict "Approved but merge skipped: <merge_skip_reason>".
- `is_self == true` â†’ `--comment` (GitHub blocks self-approval; auto-merge still works), verdict "Approved & queued for merge (posted as comment â€” self-PR)" or "Approved but merge skipped: <merge_skip_reason> (posted as comment â€” self-PR)".

Post the review with the chosen action and verdict baked into the body from Step 5:

```bash
gh pr review <pr> <--approve|--comment> --body "$(cat <<'EOF'
<review body from Step 5, verdict line filled in>
EOF
)"
```

If `should_merge == true`, queue the merge:

```bash
gh pr merge <pr> --auto --merge
```

`--auto` makes GitHub merge the PR once required checks pass. `--merge` uses a merge commit (not squash, not rebase).

If `gh pr merge --auto` fails with `auto-merge is not enabled for this repository`, the repo's settings â†’ "Pull Requests" â†’ "Allow auto-merge" checkbox is off. Report the failure to the user along with the verdict, but do NOT attempt a non-auto merge as a fallback (the user may have other reasons for wanting a human gate).

### 6b. Issues path (â‰¥1 critical or important finding)

Pick the review action based on `is_self`:

- `is_self == false` â†’ `--request-changes`, verdict "Changes requested".
- `is_self == true` â†’ `--comment` (GitHub blocks self-request-changes; flag in the verdict so the reader isn't confused by the milder review type), verdict "Changes requested (posted as comment â€” GitHub blocks request-changes on own PR)".

```bash
gh pr review <pr> <--request-changes|--comment> --body "$(cat <<'EOF'
<review body from Step 5, verdict line filled in>
EOF
)"
```

Do NOT merge.

---

## Step 7: Restore the Working Tree (cleanup)

This step runs as a `finally`-style cleanup on every exit path after Step 2 (`gh pr checkout`) succeeds â€” including the abort paths in Step 3 (sub-skill failure) and any error in Steps 4-6. If Step 2 itself failed, skip this step (no checkout happened, nothing to restore).

Switch back to the branch saved in Step 1:

```bash
git checkout <original-branch>
```

This avoids leaving the user stranded on the PR's head branch.

---

## Step 8: Final Report

Print a brief summary to the user:

- Verdict â€” use the same string posted to GitHub (one of: `Approved & queued for merge`, `Approved but merge skipped: <reason>`, `Changes requested`, plus the `(posted as comment â€” ...)` suffix on self-PRs).
- Counts: `critical / important / suggestion`.
- URL of the posted review (returned by `gh pr review`).
- Original branch restored.
